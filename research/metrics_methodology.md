# Методология метрик

Дата ревизии: 2026-05-08.

## Метрики отзывов

Все метрики отзывов считаются по строкам `data/processed/reviews_labeled.csv`. Одна строка - один сохраненный review-evidence фрагмент.

| Метрика | Как считаем | Как читать |
|---|---|---|
| `total_reviews` | Число строк в `reviews_labeled.csv` | Размер seed-выборки |
| `queue_related_share` | `mentions_queue == True` / `total_reviews` | Доля строк, где есть очередь или ожидание |
| `negative_share_among_queue_related` | `sentiment == negative` среди queue-related / queue-related | Насколько часто упоминание очереди негативное |
| `price_related_share` | `mentions_price == True` / `total_reviews` | Доля строк про цену, билет, доплаты или экспресс |
| `crowd_related_share` | `mentions_crowd == True` / `total_reviews` | Доля строк про толпу или большое число людей |
| `fastpass_related_share` | `mentions_fastpass == True` / `total_reviews` | Доля строк про экспресс, VIP, браслет или fastpass |
| `attractions_mentioned_count` | Счетчик названий из `attractions_mentioned` | Какие аттракционы явно названы в текстах |
| `frequent_queue_terms` | Частотность слов в queue-related текстах после стоп-слов | Какие слова чаще всего идут рядом с проблемой очередей |

Разметка сейчас rule-based. Это нормально для MVP, но не является ML-классификатором. После роста выборки до 100-300 строк нужно проверить словари вручную и посчитать precision/recall на небольшой размеченной вручную подвыборке.

## Метрики симуляции

Симуляция - это сценарная модель, а не реальные операционные данные парка. Текущая версия моделирует выбранные аттракционы, а не весь парк целиком. Поэтому в конфиге есть `selected_attractions_demand_share`: доля общего ride demand, которая попадает в выбранный набор аттракционов.

| Метрика | Как считаем | Как читать |
|---|---|---|
| `avg_wait_time` | Среднее по всем samples ожидания за прогон | Среднее модельное ожидание в минутах |
| `max_queue_length` | Максимум очереди по всем аттракционам и шагам времени | Насколько большим становится худший bottleneck |
| `queue_variance` | Дисперсия очередей между аттракционами | Чем выше, тем сильнее перекос нагрузки |
| `attractions_completed_per_visitor` | Завершенные посещения аттракционов / число гостей | Сколько аттракционов в среднем успевает гость |
| `paid_slots_sold` | Число купленных быстрых слотов | Объем платного priority inventory |
| `revenue_generated` | `paid_slots_sold * avg_slot_price` | Валовая выручка от fast slots |
| `satisfaction_score` | Индекс 0-100 из функции `satisfaction` | Условная UX-оценка для сравнения сценариев |

## Что стало реалистичнее

- Спрос распределен по дневному профилю, а не равномерно.
- `target_attractions_per_guest` задает желаемое число ride attempts.
- `operational_capacity_factor` снижает теоретическую capacity до полезной модельной capacity.
- Paid fast slots ограничены через `max_fast_slot_capacity_share`.
- Спрос на paid fast slots появляется только при заметном ожидании.
- Финмодель берет wait time, sold slots и revenue из симуляции, а не из ручных констант.

В текущем JSON поле называется `visitor_satisfaction_score`. В презентационном тексте можно называть его `satisfaction_score`, но важно помнить, что это model output.

## Что нельзя утверждать

- Нельзя говорить, что текущие проценты отражают всех гостей парка.
- Нельзя сравнивать модельные минуты ожидания с фактическими минутами без операционных данных.
- Нельзя смешивать review-evidence и model outputs в один «доказанный факт».

Корректная формулировка: «В seed-выборке есть повторяющиеся упоминания очередей, а симуляция показывает, что при заданных assumptions перераспределение спроса снижает перекос нагрузки и создает fast-slot revenue».
