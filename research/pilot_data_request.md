# Pilot Data Request

Dream Queue should start with the minimum operational data needed to validate a narrow pilot. The recommended first scope is 4-8 weeks, 5-8 bottleneck attractions, and peak windows only.

## Minimum Data Request

| Data | Why it is needed | Sensitivity |
|---|---|---|
| Hourly attendance by day type | Identify peak windows and demand shape. | Medium |
| Attraction capacity: seats, cycle time, dispatch interval | Estimate true throughput and slot inventory. | Low/medium |
| Wait-time logs or manual observations | Identify bottlenecks and validate baseline. | Medium |
| Downtime and temporary closure logs | Add operational realism to simulation. | Medium |
| Existing express/VIP demand and constraints | Understand current priority-access economics and fairness limits. | High |
| F&B/retail revenue by hour or zone, aggregated | Estimate whether freed wait time has a monetization proxy. | High |
| App/site/ticketing capabilities | Scope QR, ticket binding, kiosk, and integration feasibility. | Medium |
| Queue complaints, NPS, or review tags | Validate guest pain and pilot success metrics. | Medium |
| Park map and walking-time estimates | Make routing realistic. | Low |
| Staff constraints at attraction entrances | Define validation SOP and exception handling. | Medium |

## Non-Sensitive Fallback

If commercial or operational data cannot be shared at first, start with:

- manual wait-time observations;
- estimated attraction capacity;
- counts at selected ride entrances;
- simulated slot inventory;
- QR validation prototype;
- before/after guest survey;
- operator interviews about peak-window pain and staff constraints.

## Pilot Scope

- 4-8 weeks.
- 5-8 bottleneck attractions.
- Peak days and peak windows only.
- Baseline measurement before optimization claims.
- Simple VQ slots.
- Limited paid priority cap if the park wants to test monetization.
- Route recommendations to nearby attractions, shows, F&B, and retail.
- Operator dashboard.
- QR validation prototype.
- Staff SOP for late arrivals, no-shows, wrong group, downtime, and app/kiosk exceptions.

## Decision Metrics

Scale the pilot only if at least 2-3 of these improve without unacceptable fairness tradeoffs:

- maximum queue on selected rides;
- wait-time predictability;
- guest complaints about queues;
- slot utilization;
- no-show handling;
- operator visibility;
- F&B/retail proxy during wait windows;
- paid priority revenue without unacceptable standby degradation.

All pilot metrics must be measured before/after or clearly labelled as survey/model outputs.
